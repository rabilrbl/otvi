## MODIFIED Requirements

### Requirement: Proxy access must be constrained to server-issued stream contexts
The system MUST reject proxy requests that are not tied to an active server-issued stream context. The proxy MUST only resolve upstream targets and forwarding behavior from context data created by trusted backend stream resolution, and it MUST NOT act as a general-purpose fetch endpoint for caller-supplied URLs. This constraint applies equally to HLS segment/key proxy requests, DASH segment/manifest proxy requests, and DRM license proxy requests.

#### Scenario: Request without valid stream context is rejected
- **WHEN** a client calls the proxy endpoint without a valid active context token
- **THEN** the system returns a client error and does not fetch any upstream content

#### Scenario: Proxy request is limited to context-approved upstream target
- **WHEN** a client uses a valid stream context token generated by the backend stream endpoint
- **THEN** the proxy fetches only the upstream target derived from that context and applies only the headers, cookies, and rewrite rules stored in that context

#### Scenario: DRM license proxy request without valid context is rejected
- **WHEN** a client sends a POST to `/api/proxy/drm/{token}` with a token that does not exist in the proxy context cache
- **THEN** the system returns a client error and does not forward any license request to an upstream server

#### Scenario: DRM license proxy resolves upstream target from context only
- **WHEN** a client sends a valid DRM license proxy request
- **THEN** the upstream license URL is derived solely from the `ProxyContext` DRM license URL field, not from any client-supplied URL parameter
