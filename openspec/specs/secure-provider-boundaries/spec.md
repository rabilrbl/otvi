# Secure Provider Boundaries

## Requirements

### Requirement: Proxy access must be constrained to server-issued stream contexts
The system MUST reject proxy requests that are not tied to an active server-issued stream context. The proxy MUST only resolve upstream targets and forwarding behavior from context data created by trusted backend stream resolution, and it MUST NOT act as a general-purpose fetch endpoint for caller-supplied URLs.

#### Scenario: Request without valid stream context is rejected
- **WHEN** a client calls the proxy endpoint without a valid active context token
- **THEN** the system returns a client error and does not fetch any upstream content

#### Scenario: Proxy request is limited to context-approved upstream target
- **WHEN** a client uses a valid stream context token generated by the backend stream endpoint
- **THEN** the proxy fetches only the upstream target derived from that context and applies only the headers, cookies, and rewrite rules stored in that context

### Requirement: Provider route authorization must be enforced consistently
The system MUST apply the same provider authorization policy across provider authentication, channel listing, category listing, and stream resolution routes. For per-user providers, the policy MUST enforce the authenticated user's provider allow-list. For global providers, the policy MUST enforce administrator-only credential management semantics.

#### Scenario: Restricted user is blocked from non-listed provider route
- **WHEN** an authenticated user attempts any provider-bound route for a provider that is not in that user's allowed provider list
- **THEN** the system returns a forbidden response before any provider session lookup or upstream request occurs

#### Scenario: Non-admin user is blocked from global provider credential operations
- **WHEN** a non-admin user attempts provider authentication or logout for a global-scope provider
- **THEN** the system returns a forbidden response and does not modify stored provider credentials

### Requirement: Provider session state must not leak across users or providers
The system MUST isolate outbound provider session state so cookies, headers, and credential-derived context for one user or provider cannot be reused implicitly by another user or provider through shared HTTP client state.

#### Scenario: Session state from one user does not affect another user's provider request
- **WHEN** one user authenticates with a per-user provider and another user accesses the same provider without authenticating
- **THEN** the second user's request does not inherit cookies or session state from the first user

#### Scenario: Session state from one provider does not affect another provider
- **WHEN** one provider interaction stores cookies or request state for an upstream domain
- **THEN** requests for a different provider do not reuse that stored state unless it was explicitly created for that provider context

### Requirement: Internal provider and proxy failures must be redacted at API boundaries
The system MUST avoid returning raw internal exception text, raw upstream response bodies, or sensitive proxy/header/cookie values to API clients or routine logs. It MUST expose stable error messages suitable for clients while retaining enough server-side observability for debugging.

#### Scenario: Upstream provider failure returns sanitized API error
- **WHEN** a provider request fails because of an upstream error or malformed response
- **THEN** the client receives a sanitized error response that does not include raw upstream response bodies, secrets, or internal stack details

#### Scenario: Proxy debugging does not log sensitive material by default
- **WHEN** the proxy handles requests that include resolved headers, cookies, or manifest-derived credentials
- **THEN** routine logs do not emit those sensitive values in plaintext
